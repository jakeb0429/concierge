# Concierge deep review — 2026-07-08

Methodical review of UX, performance, features, and correctness after the five-phase
multi-user build, run from three perspectives. Three parallel review passes (bug hunt,
performance, security/authorization) plus a hands-on UX walkthrough; every finding below
was **verified against the actual code**, and everything marked ✅ is **fixed, tested, and
deployed** (commit `6442863` + follow-ups). Items marked ▶ are recommendations left for
a decision.

---

## 1. Executive perspective — "is this a client-ready product?"

**Verdict: yes, with today's fixes it crossed the bar.** The review found the platform
functionally complete for its promise (email in → routed → grounded human-confirmed reply
→ self-improving Brain), but it had three classes of executive-level risk that are now closed:

**Multi-tenant isolation (the one that mattered most).** Before today, any signed-in user
of one brand could — with a guessed/leaked record id — read the other brand's tickets,
customer data, and attachments, and even **send outbound email from the other brand's
mailbox**. Five routes/pages were affected (send-confirm, draft-generation, attachments,
ticket page, customer page). All now hard-scope every lookup to the session's tenant. ✅
With Stingray onboarding real users soon, this was found at exactly the right time.

**Governance of the Brain.** The knowledge base is the product's core asset, and it was
editable by any signed-in agent, manager reviews could be self-approved, and an admin
typo in "Add teammate" could silently demote the super-admin. Now: direct Brain edits and
review approvals require team_lead+; agents change the Brain only through their gated
training queue; signal resolution is limited to leads/admins or the routed specialist;
add-teammate is create-only. ✅

**Cost integrity.** A one-character-class bug (Postgres jsonb reorders JSON keys) meant the
AI customer-read cache **never hit** — a Sonnet call on every ticket/profile/draft view,
silently. Fixed with field-level comparison; also the read no longer blocks page renders
(it regenerates after the response). ✅ This pattern — verify cache hit rates, don't assume —
is written into the handoff.

**▶ Executive recommendations (not built, worth deciding):**
- **Response-time KPIs**: time-to-first-reply and resolution-time per assignee/category on
  /analytics — the data (AuditEvents + message timestamps) already exists.
- **Weekly digest email** to admins: volume, unassigned backlog, training pending, expired notes.
- **Audit-trail UI**: AuditEvent is complete but has no viewer; a read-only admin page would
  help client trust ("who approved this Brain change?").
- **Backups/DR statement**: Supabase PITR covers the DB, but a documented recovery runbook
  (docs/HANDOFF has runbooks; a client-facing one-pager doesn't exist yet).

## 2. Operational / triage-admin perspective — "can hello@ run the floor on this?"

**The queue is now trustworthy.** Two bugs would have actively lied to the triage admin:

- **Reply states were wrong across week/month boundaries** — messages sorted alphabetically
  by weekday name ("Fri" < "Sun"), so a rep-answered thread could show "follow-up / waiting
  Nd" and sort into the answer-first queue, and vice versa. This poisoned the importance
  ordering the whole triage view is built on. Fixed to true chronological order. ✅
- **Urgent tickets could silently fall off** the 100-row window once volume grows; urgent
  is now fetched separately and can't be displaced. ✅
- **Customer replies to resolved tickets vanished** — they landed on the ticket but the
  ticket stayed `resolved`, outside every working view. Intake now reopens it (status →
  new, audited `ticket_reopened`). ✅
- **The "My tickets" tab count disagreed with its own list** (different status filters);
  now computed from the identical query. ✅
- Case-variant sender emails no longer fork one customer into two histories (0 existing
  duplicates verified before the fix). ✅

**▶ Triage recommendations:** the workload strip counts all open assigned tickets, including
"waiting on customer" — fine for load, but a needs-reply-only number would better reflect
pressure. And reassignment reloads the page (functional, slightly clunky) — a toast-style
inline update is a polish item.

## 3. Individual-rep perspective — "is the reply page fast and honest?"

- **Ticket page no longer stalls on the AI.** The customer read used to block the render
  for 2–6s whenever the customer had new activity — which is exactly when a rep opens the
  ticket. It now always serves instantly from cache and refreshes in the background. ✅
- **The inbox got ~90% lighter** — it was shipping every message's full body for 100
  tickets (megabytes per load); now only directions/timestamps plus one truncated snippet
  per ticket. The thread view also stops fetching the never-rendered HTML bodies. ✅
- **The review gate is honest now**: regenerating an approved draft used to keep the green
  "manager approved" light on brand-new unreviewed text (and a regenerated pending draft
  stayed locked). Draft-generation errors now surface as a message instead of silently
  corrupting the editor. A ticket resolved without replying no longer claims "Reply sent." ✅
- Note-expiry dates no longer display/expire a day early in Eastern time. ✅

## 4. Testing & debugging

- Every fix typechecked (`tsc` clean) and the app was exercised end-to-end locally with three
  real sessions (admin, cross-brand admin, agent): sign-in, inbox views per role, ticket
  workspace, Brain-edit rejection for agents (403 verified live), brand switcher **both
  directions** (found jake@scribechs.com had no Rheos row — provisioned, now switches cleanly).
- Production verified after deploy: password login (302→session→200), full magic-link chain
  (request → email → verify `magic=verified` → authenticated 200), live intake run on the
  server, cron intact.
- `scripts/e2e-login-test.cjs` hardened: it now sends its own link request and only accepts
  the email it caused — pre-existing fresh emails carry superseded tokens and produced
  false failures (this bit me during verification; it will not bite you).
- **▶ Recommendation:** a real test suite is the next maturity step — the highest-value
  targets are triage classification (fixture emails → expected category), reply-state, and
  route authorization (agent vs admin vs cross-tenant). Currently the platform relies on
  typecheck + e2e scripts + this kind of review.

## 5. Performance changes shipped (summary)

| Change | Effect |
|---|---|
| Customer read: cached serve + background regen | ticket/profile pages never wait on the model |
| Insight cache actually hits (jsonb key-order bug) | eliminates a hidden Sonnet call per view |
| Inbox: no message bodies; slim customer/channel selects | ~90% smaller inbox payload |
| Ticket thread: drop unused `html` column | threads with long emails load several× lighter |
| `getCurrentTenant`/`sessionUser` request-deduped | ~2-3 fewer pooler round-trips per page |
| detect-learning: batched citations + watermarks | no N+1; resolved proposals can't resurrect |
| New indexes: CustomerOrder(tenantId,source), Ticket(customerId), AuditEvent(tenantId,action), Draft(tenantId,status), User(email) | sources panel, profile, detector, analytics all indexed |
| Profile lists bounded (take 100) | wholesale accounts can't blow up the page |

# Round 2 — post-feature UX pass (same day, after digest/audit/KPIs/backups shipped)

**Implemented in this pass:**
- Brain manager UI now matches the API gates: agents see entries and their own
  assigned training questions but no Add/Edit/Approve affordances (leads/admins
  unchanged). Signal cards show buttons only to leads/admins or the assigned specialist.
- Workload strip shows **pressure, not just load**: "Name · 2 to answer / 5 open"
  (amber when someone has tickets awaiting a reply), sorted by pressure.
- Reassigning from the inbox is a soft refresh now (keeps scroll position).

**New recommendations (round 2, in rough priority):**
1. **Nav grouping** — admins now carry 9 top-level links; an "Admin" cluster
   (Digest/Team/Sources/Audit) or overflow menu is warranted at the next UI touch.
2. **SLA targets** — the response-time KPIs beg for a target line ("first reply
   under 4 business hours") with breach highlighting on /analytics and the digest.
   Needs Jake to pick the numbers.
3. **Routing self-tuning** — when the admin repeatedly reassigns category X to the
   same person, suggest adding that specialty on /users (the audit trail already
   captures every reassignment, so this is a query + a banner).
4. **Digest sender identity** — reports currently send from the justforfun Mailgun
   domain; a concierge@scribechs.com sending domain would look right for a
   client-facing product (Mailgun domain + DNS records).
5. **Rep flow** — a "next ticket →" button after send (keeps reps in the queue
   instead of bouncing to the inbox), and keyboard shortcuts later.
6. **Off-box backup copy** — nightly dump currently lives on the app box only
   (documented in BACKUP-RECOVERY.md); rsync to a second host or object storage
   is the next hardening step.
7. **Token-spend visibility** — a daily spend estimate in the digest once usage
   reporting is wired (console.anthropic.com is the manual source today).
